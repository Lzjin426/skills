# 中文技术文档 AI 痕迹检测与改写清单

专门针对软著申请的**用户手册 / 设计说明书**。这类文档的 AI 味和小说散文不一样——
多的是"总分总三段式"、"对称工整的并列"、"模板化的功能描述"。

如果你已经装了 `humanizer-zh` skill，那是通用去 AI 味的文字编辑；本清单**专注技术文档**，
侧重术语一致性、功能对应性、图文并茂的表达感。

---

## ❗ 红线（软著文档硬要求）

| 模式 | 处理 |
|------|------|
| 截图里的真实用户名 / 公司名 / 真实数据 | 马赛克替换（但软著又禁止马赛克——实际做法是截图前先用假数据） |
| 截图里的日期晚于申请表完成日期 | 换截图 |
| 软件名、版本号与申请表不一致 | 全文替换为申请表里的名字 |
| 描述了但代码里没实现的功能 | 删除该节或在代码里补上 |
| 代码里有但说明书没讲的核心功能 | 补说明书 |

**命名一致性是最高频出错点**。用 Grep 在文档和代码页眉、申请表、封面之间交叉验证
软件全称和版本号。

## 🔴 一级痕迹（AI 口癖，高密度出现极易识别）

### 1. AI 高频口头禅

下面这些词/短语在 AI 中文输出里出现频率远高于真人技术写作：

| AI 口癖 | 建议替换 |
|---------|----------|
| 综上所述 / 总的来说 / 总而言之 | 删掉，或改成"所以"/"因此" |
| 值得注意的是 / 需要指出的是 | 直接说要点 |
| 不仅……而且…… | 拆成两句 |
| 首先……其次……然后……最后…… | 去掉排比骨架，按实际步骤写 |
| 本软件（每段都用） | 偶尔用、偶尔省略主语、偶尔用"软件"/"系统"替换 |
| 旨在…… / 致力于…… | "用来"/"目的是" |
| 通过……实现…… | "用 X 做 Y" |
| 在 XX 的基础上 | 删掉或"基于 XX" |
| 一方面……另一方面…… | 拆两句 |
| 具有……特点 / 特征 | 直接描述 |
| 极大地 / 显著地 / 有效地 | 删掉（这些副词是浮夸信号） |
| 赋能 / 助力 / 打造 / 构建 | 换成"做"/"建"/"写" |
| 全方位 / 多维度 / 一体化 / 高效 / 智能化 | 几乎都是套话，能删就删 |

**扫描方法**：用 Grep 搜上面这些词，逐处决定删除 / 改写 / 保留（保留少量即可）。

### 2. 对仗工整的三段式

**AI 特征**：功能介绍几乎都是三段——"功能说明 / 操作步骤 / 注意事项"，每段字数差不多、
句式对仗。

```markdown
### 数据导入功能

本功能旨在为用户提供便捷的数据导入能力。用户可以通过点击界面上的"导入"按钮，
选择本地文件进行上传。系统会自动解析数据格式并将其存入数据库。

**操作步骤**：
1. 点击主界面右上角的"导入"按钮
2. 在弹出的文件选择框中选择目标文件
3. 点击"确认"按钮
4. 系统自动完成数据导入

**注意事项**：
- 支持 .csv 和 .xlsx 两种格式
- 单次导入文件大小不超过 50MB
- 导入过程中请勿关闭窗口
```

**真人风格**：

```markdown
### 数据导入

"导入"按钮在主界面右上角，点开选本地文件就行。支持 csv 和 xlsx，文件别超过 50MB。
导入过程中不要关窗口，中途断了得重来一遍——这个后面版本再优化。

![导入界面截图](imgs/import.png)
```

特征：
- 句子有长有短
- 骨架不对称（先说入口、然后说限制、最后加个吐槽式的 TODO）
- 少量"后面版本再优化"这种时间维度

### 3. 过度 bullet

**AI 特征**：能用列表绝不用段落，每节都是若干 bullet。

**改写**：
- 描述性的内容改回段落
- 真正并列的清单（配置项、文件格式）保留 bullet
- 原则：bullet 密度控制在整篇文档的 30% 以下

### 4. 模板化的章节结构

**AI 特征**：每个功能模块小节都严格按"简介 / 界面截图 / 操作说明 / 参数说明 / 注意事项"
五段走，不多不少。

**改写**：
- 简单功能砍到 2–3 段
- 复杂功能展开得更细，允许 5–6 段
- 不是每个功能都配"注意事项"——没必要就删

### 5. 过度形式化表述

| AI | 真人 |
|----|------|
| 用户可以通过本软件实现对数据的管理 | 软件管数据 / 用来管数据 |
| 该功能为用户提供了便捷的操作体验 | 这个按钮点一下就能做 X |
| 系统将自动对输入数据进行校验 | 输进去之后系统会校验 |
| 在使用本功能之前，请确保已安装 X 依赖 | 用之前装好 X |

## 🟡 二级痕迹（风格偏形式化）

### 6. 过度使用被动语态

中文技术文档的 AI 味很大一块是"……被处理"/"……被保存"/"……将被展示"的被动结构。
中文母语者更习惯主动或省略主语。

```markdown
# AI
用户的输入数据将被系统进行校验，校验结果将被展示在界面上。

# 改后
系统会校验输入数据，校验结果显示在界面上。
```

### 7. 长定语堆叠

**AI 特征**：`基于用户行为数据分析的个性化推荐功能模块`——一长串定语修饰一个名词。

**改写**：拆成短句。`根据用户行为数据做个性化推荐。` 简单直接。

### 8. "本文 / 本节 / 本章"滥用

AI 文档每段都以"本节介绍……"/"本章将讲解……"开头。真人技术文档偶尔有，但不会每节都来一遍。

**改写**：删掉大多数，保留 1–2 处作为正式文档的骨架感。

### 9. 结尾收束套路

AI 文档习惯在每节末尾写"以上即为 X 功能的说明"/"本章节介绍了……"。

**改写**：直接结束，不要收束。

## 🟢 三级纹理（加分项）

### 10. 版本迭代痕迹

在说明书里偶尔出现：

- "（V1.0 暂未支持，后续版本会加）"
- "注：这个在 Windows 7 上测过，但 macOS 10.14 以下没测"
- "早期版本用的是 JSON，V1.0 起改成 SQLite 存储"

这种"作者知道产品历史"的语气是 AI 很难伪造的。**约束**：全文档最多 2 处，不要集中出现在同一章节。

### 11. 真实使用场景描述

AI 写的"典型使用场景"往往抽象套话（"适合各类用户进行高效数据处理"）。真人写的场景具体：

```markdown
# AI
本软件适合各类工程技术人员进行参数优化工作，能够有效提升工作效率。

# 改后
我平时拿这个来做齿轮箱接触应力的初步校核，原本用 Excel 公式搞，要手动改十几个单元格。
现在改一下参数点一下就出结果。
```

### 12. 图文错位 / 不完美

- 截图顺序和正文描述顺序**小幅错开**——真人写完文字再配图时会对不上
- 偶尔出现"如下图所示"但图在下一页
- 图片名字不统一 `screenshot1.png` / `界面.png` / `img_home.jpg` 混着来

**约束**：全文档最多 1 处。太多就乱。

### 13. 术语混用

同一个概念偶尔切换说法：
- "数据库" / "库" / "SQLite 文件"
- "用户" / "使用者" / "人"
- "导入" / "上传" / "加载"

AI 倾向全文统一术语，真人经常切来切去。**约束**：每个概念允许 2 种说法（不要超过 3 种），
按场景切。

## 改写检查表（文档 rewrite 收尾必看）

- [ ] 软件全称、版本号与申请表一致（grep 验证）
- [ ] 页眉格式统一：`软件全称 + 版本号 + 页码`
- [ ] 封面、目录齐全
- [ ] 每个核心功能都配有截图（文字为主，图片不占整页）
- [ ] 截图里无真实用户数据、无真实公司名、无 `XX 银行` 之类易被挑刺的命名
- [ ] 截图日期 ≤ 申请表完成日期
- [ ] 说明书里提到的**每个功能**在代码里能找到对应模块/函数（程序-文档对应性校验）
- [ ] 总页数 ≥ 15（主流实践 15–30 页更稳）
- [ ] 已扫一遍 AI 口癖清单（第 1 条那个表格）并清理
- [ ] 生成 diff 记录 `dehumanize-diff-doc-<timestamp>.md`

## 一段完整改写示例

**改前（典型 AI 风格）**：

```markdown
## 3.2 参数优化模块

### 3.2.1 功能概述

本模块旨在为用户提供一套完整的参数优化解决方案。通过内置的多种优化算法，
用户可以针对不同的工程场景选择合适的算法，从而实现对设计参数的高效寻优。
本模块不仅支持单目标优化，而且支持多目标优化，具有灵活性高、易用性强的特点。

### 3.2.2 操作步骤

1. 首先，用户需要在主界面点击"参数优化"按钮进入本模块
2. 其次，在参数配置界面填写待优化的参数范围
3. 然后，选择优化算法（目前支持 GA、NSGA-II、PSO 三种）
4. 最后，点击"开始优化"按钮，系统将自动执行优化流程

### 3.2.3 注意事项

- 优化过程可能耗时较长，请耐心等待
- 建议合理设置种群大小以提升优化效率
- 本模块的计算结果仅供参考
```

**改后**：

```markdown
## 3.2 参数优化

点"参数优化"进来就是这个模块，用来在给定参数范围里找优化解。目前支持 GA、NSGA-II、
PSO 三种算法，前两个用得多，PSO 是后加的。

单目标、多目标都能跑，多目标用 NSGA-II。

![参数优化界面](imgs/optim.png)

操作上不复杂：

- 参数范围填在上面的表格里，每行一个变量
- 下拉选算法
- 点"开始优化"

种群数量默认 50，大数据集调到 100–200。跑起来之后日志会滚屏显示当前代数和最优值，
大一点的问题跑个几分钟很正常。结果出来在右侧的 Pareto 前沿图上（多目标场景）。

注：PSO 在 V1.0 还没做收敛判据，手动看着停。后续版本补。
```

改动点：
- 删掉"旨在……为用户提供"/"本模块不仅……而且……"/"具有……特点"这套 AI 骨架
- 三个对仗子标题合并成行文段落
- 加具体数字和经验值（种群 50、100–200）
- 加时间维度痕迹（"PSO 是后加的"/"V1.0 还没做"）
- 加图在文中、不是"如下图所示"模板
- 口语化一点但不失准确（"跑起来"/"看着停"）
