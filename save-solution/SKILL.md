# Save Solution

将 code agent 对话中的问题解决过程提炼为一条可检索的知识笔记，存入 Obsidian 知识库。

## 为什么需要这个 skill

用户经常借助 code agent（Claude Code、Codex 等）解决技术问题，但对话结束后这些经验容易丢失。
这个 skill 把"问题 → 解决"的过程沉淀为结构化笔记，让知识可复用、可检索。

## 工作流程

### Step 1：回顾对话，判断复杂度，提取信息

**首先判断这是简单问题还是复杂问题**，这决定笔记的结构：

| 信号 | 说明 |
|------|------|
| 尝试了多个方案才成功 | 对话中出现了"不行""报错""换个方法"等反复 |
| 涉及多个系统/工具的交互 | 问题横跨多个组件，不是单点修复 |
| 需要先理解概念才能解决 | 根因涉及对某个原理/机制的认知空白 |
| 对话轮数多、中间有明显弯路 | 花了较长时间才收敛到最终方案 |

有**任意一条**信号，视为复杂问题，使用扩展模板（见 Step 2）。

提取原则：
- 简单问题聚焦最终方案，弯路不记
- 复杂问题的弯路和失败尝试有教训价值，单独记录
- 代码片段只保留关键部分，不要整段复制
- 用简洁的中文书写，技术术语保留英文原文（如 SSH、API、Docker）

### Step 2：生成笔记文件

**文件名**：用一个简短精炼的关键词短语命名，能让人一眼看出是什么问题。
- 好的命名：`SSH 连接 Mac 配置.md`、`Tauri 签名密钥生成.md`、`MATLAB 编译 Python 调用.md`
- 避免：过长的句子、日期前缀（Obsidian 有创建时间元数据）、模糊的名字如 `问题记录.md`

**文件路径**：`/Users/lzjin/Fullstop_base/0-inbox/<文件名>.md`

**简单问题模板**（四区块）：

```markdown
# 问题描述

<用 1-3 句话简明扼要地描述遇到了什么问题>

# 环境与上下文

- <项目/工具/语言/OS 等，用列表形式>

# 解决方案

<关键步骤，用有序列表或代码块>

# 根因分析

<为什么会出现这个问题，1-2 句话>
```

**复杂问题模板**（六区块，在四区块基础上增加"踩过的坑"和"关键概念"）：

```markdown
# 问题描述

<比简单问题更详细——描述问题的表现、影响范围、最初的错误方向>

# 环境与上下文

- <项目/工具/语言/OS 等，用列表形式>

# 踩过的坑

- **方案 A**：XXX → 失败原因（为什么行不通）
- **方案 B**：XXX → 失败原因

# 关键概念

- <解决这个问题需要理解的核心知识点或原理，1-3 条>

# 解决方案

<最终有效的关键步骤，可以比简单问题更详细>

# 根因分析

<复杂问题的根因往往更深——说清楚认知盲点在哪，下次如何更早识别>
```

内容要求：
- 每个区块都要有实质内容，不要留空或写占位符
- 解决方案部分如果涉及命令或代码，用 code block 包裹并标注语言
- 简单问题控制在 30 行内；复杂问题不限，但每个区块只写有价值的内容，不要凑字数

### Step 3：在当天日笔记中添加链接

日笔记路径规则：`/Users/lzjin/Fullstop_base/03-note/<年>/Daily/<月>月/<年>-<月>-<日>.md`

例如 2026 年 4 月 12 日 → `/Users/lzjin/Fullstop_base/03-note/2026/Daily/4月/2026-04-12.md`

注意月份不补零：1月、2月 ... 12月。

操作方式：
1. 读取当天日笔记
2. 找到 `# 记录` 区块
3. 在该区块末尾追加一行：`- [[<笔记文件名（不含 .md）>]]`
4. 如果日笔记不存在或没有 `# 记录` 区块，在文件末尾追加 `\n# 记录\n\n- [[<笔记文件名>]]`

### Step 4：确认完成

告诉用户：
- 笔记已保存到哪个路径
- 日笔记中已添加链接
- 简要预览笔记的核心内容（2-3 句话）
