# 投稿级中文润色规则

本文件服务于“保留原意、提升学术表达、压实论证”的润色任务。

## 1. 基本原则

- 先判断逻辑是否成立，再润色
- 不擅自发明数据、实验、引用、结论
- 不把模糊问题通过修辞掩盖
- 不把工程常识写成已验证事实

## 2. 高价值改写方向

### 2.1 把空话改成问题指向

低质量：
- 本文具有重要意义
- 该方法效果良好

高质量：
- 针对现有方法在某场景下的某类不足，本文提出……
- 试验结果表明，在某指标上较某基线提升……

### 2.2 把“做了什么”改成“为何这样做”

低质量：
- 设计了模块 A、模块 B 和模块 C

高质量：
- 为解决某问题，设计模块 A；为进一步处理某限制，引入模块 B；最终通过模块 C 完成……

### 2.3 把强断言压回证据边界

低质量：
- 本文方法具有很强的泛化能力

更稳妥：
- 在所测试的多个场景/数据集上，本文方法表现出较好的泛化能力

### 2.4 把并列堆砌改成递进链条

低质量：
- 首先……其次……再次……最后……

更稳妥：
- 基于上述分析……
- 在此基础上……
- 进一步地……
- 因而……

不要机械堆叠连接词。重点是让语义递进成立。

## 3. 分章节润色重点

### 3.1 题目

- 准确呈现对象、方法、任务
- 避免口号式、副标题式、评价式措辞
- 除非领域习惯明确，不轻易使用“新型”“高效”“先进”

### 3.2 摘要

必须同时具备：
- 问题
- 方法
- 关键机制
- 定量结果
- 应用价值

摘要常见修法：
- 删除过长背景
- 压缩常识性铺垫
- 补出方法机制的关键名词
- 把“效果较好”替换成数字

### 3.3 前言

- 文献回顾按类别组织，不按作者流水账组织
- 每类文献后面要落到具体不足
- 最后一段明确本文工作的针对性

### 3.4 方法

- 先给总框架，再给子模块
- 每个模块写清输入、操作、输出、目的
- 变量和符号首次出现即定义

### 3.5 结果分析

- 不能只报数值，要解释变化原因
- 优先写“哪项设计导致何种变化”
- 不把偶然现象包装成普遍规律

### 3.6 结论

- 回收贡献，不重复背景
- 用正文证据支撑每一点
- 展望要克制，写成“后续可进一步……”，不要写成“必将……”

## 4. 中文工科表达禁区

避免：
- 极其、非常、显著领先、全面优于、完全解决
- 口语化主观判断
- 过度宣传式意义拔高
- 模糊主语，如“其”“该方法”连续出现导致指代混乱

推荐：
- 针对……，提出……
- 为验证……，开展……
- 结果表明……
- 与……相比……
- 说明/验证了……

## 5. 术语与格式统一

检查并统一：
- 中英文术语首次出现格式
- 缩写首次定义
- 图、表、式、章节引用
- 变量字体和符号命名
- 同义词混用，例如“方法/模型/策略/框架/系统”是否真的同义

## 6. 润色交付要求

若只做语言优化，输出修改稿即可。

若原文存在明显论证缺口，先给一句判断：
- “以下修改提升表达，但不能替代补实验/补论证。”

然后再给润色版本，避免让作者误以为已经解决研究设计问题。
