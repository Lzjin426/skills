# Profile: KB Article

## When to use

Use this profile for durable knowledge-base content that should still make sense when revisited later.

Before drafting, identify which of these four jobs is dominant:
- concept explanation
- comparison
- result interpretation
- method/process

## Reader expectation

The reader expects to understand the topic quickly, picture what it refers to, and return later without needing the original conversation. The tone should stay professional, neutral, and reusable.

## Default structure

Choose one stable skeleton instead of inventing a fresh outline each time.

- **Concept explanation**
  - what this thing is
  - what it looks like in practice
  - what it is doing
  - what its boundary is
  - how to place it back into the current project
- **Comparison**
  - the judgment framework
  - what each side looks like in operation
  - what problem each side is solving
  - representative models, cases, or categories
  - why the distinction matters
- **Result interpretation**
  - what the result looks like
  - what the result is expressing
  - how to read it correctly
  - common misreadings
  - what role it plays in the current analysis flow
- **Method/process**
  - what problem this process solves
  - the basic flow
  - key parameters or steps
  - how to read the output
  - boundaries, errors, or failure modes
  - how to place it back into the current project

## Default carrier

Paragraph-led, supported by generous visuals, short lists, and selective tables.

## Default passes

- research: `off`
- cognitive-path: `optional`
- illustration: `encouraged`
- humanization: `off` by default

When illustration is enabled, prefer a separate illustration review step after drafting instead of relying only on the writing agent's inline judgment.

## Style defaults

- identify the article subtype before writing the outline
- explain in plain but precise language before leaning on terminology
- show the concrete thing first when readers would otherwise struggle to picture it
- optimize for future readability, not only current context
- keep a clear through-line instead of stacking disconnected mini-sections
- keep top-level sections within `3-5` by default; merge thin sections instead of scattering the topic
- make the article answer one clear judgment question instead of only accumulating facts
- keep at least one explicit visual anchor in the draft, such as what it looks like, what the output looks like, what the flow looks like, or how the comparison is shaped
- keep the tone professional, neutral, and restrained
- for Fullstop's local knowledge base, use numbered semantic headings by default, such as `1.`, `1.1`, and `1.1.1`
- prefer descriptive section titles such as "定义", "作用", "读图方法", "边界", "与脚本关系"
- avoid conversational or spoken-style headings such as "先看...", "先别管...", "你现在最该..."
- avoid tutorial-script lines such as "先记这一句", "如果你把它想成...", "你只要记住..."
