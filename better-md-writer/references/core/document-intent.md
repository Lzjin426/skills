# Better Md Writer Core: Document Intent

Use this file to identify what the document needs to do before drafting anything substantial.

## 1. Clarify the job, not the label

When the request is underspecified, identify only the missing facts that change the document:
- Who is the reader?
- What is the document for?
- What should the reader understand, decide, or do after reading it?
- Is this mainly for explanation, decision support, execution, archival, or persuasion?
- Are there format, tone, length, or template constraints?

Do not ask every question by default. Ask only what is still unknown and consequential.

## 2. Read the task across six dimensions

Use these dimensions to shape the draft:
- **Audience familiarity**: expert, mixed, newcomer
- **Purpose**: explain, compare, decide, persuade, record, instruct
- **Actionability**: informational, decision-supporting, execution-driving
- **Durability**: temporary note, reusable reference, long-lived documentation
- **Evidence load**: none, light rationale, explicit evidence
- **Structure bias**: paragraph, list, table, diagram, mixed

These dimensions matter more than forcing the request into a generic template too early.

## 3. Choose the document profile

Map the dominant job to one profile:
- **note** when the main job is quick capture or low-friction cleanup
- **kb-article** when the main job is durable explanation for later reuse
- **readme** when the main job is orientation and getting started
- **proposal** when the main job is comparing options and supporting a decision
- **tutorial** when the main job is teaching a process or concept
- **runbook** when the main job is operational execution

If two profiles both seem plausible, choose the one that best matches the reader's next action.

If the chosen profile is **kb-article**, also identify the dominant subtype before drafting:
- **concept explanation**: the reader needs to know what something is, what it looks like, and where its boundary is
- **comparison**: the reader needs a judgment framework for distinguishing multiple mechanisms, options, or families
- **result interpretation**: the reader needs to read a chart, output, metric, or analysis result correctly
- **method/process**: the reader needs to understand the input-process-output chain

This subtype choice should control the skeleton more than the author's momentary preference.

## 4. Produce a lightweight decision record

Before drafting, reduce the task to a compact internal record:

```text
Profile:
KB subtype:
Audience:
Purpose:
Actionability:
Durability:
Evidence load:
Default carrier:
Judgment question:
Visual anchor:
Clarification still needed: yes/no
```

This is a working decision aid, not something that must appear in the final document.

## 5. Stay lightweight by default

Do not enable research, diagrams, cognitive-path design, or post-processing just because the task sounds important. Enable them only if they clearly improve the result for this reader and this document.
