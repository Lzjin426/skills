# 动画审查格式

## 前/后/原因表

使用 Markdown 表格。每个问题一行。

| 之前 | 之后 | 原因 |
|---|---|---|
| `transition: all 300ms` | `transition: transform 200ms ease-out` | 指定确切的属性；避免使用 `all` |
| `transform: scale(0)` | `transform: scale(0.95); opacity: 0` | 现实世界中没有任何事物是从无到有出现的 |
| 下拉菜单使用 `ease-in` | 使用 `ease-out` 配合自定义曲线 | `ease-in` 感觉迟缓；`ease-out` 能提供即时反馈 |
| 按钮没有 `:active` 状态 | 在 `:active` 时使用 `transform: scale(0.97)` | 按钮必须对按压有响应感 |
| 弹出框使用 `transform-origin: center` | `transform-origin: var(--radix-popover-content-transform-origin)` | 弹出框从触发器位置缩放（模态框保持居中） |

## 审查清单

| 问题 | 修复方法 |
|---|---|
| `transition: all` | 针对特定属性 |
| 动画化布局属性（`width`、`height`、`top`、`left`） | 切换到 `transform` 和 `opacity` |
| UI 进入时使用 `ease-in` | 使用进入缓动：`cubic-bezier(0.22, 1, 0.36, 1)` |
| 永久设置 `will-change` | 仅在动画期间切换 |
| 起始状态为 `scale(0)` | 使用 `scale(0.85–0.95)` 配合 `opacity: 0` |
| 悬停效果缺少触摸设备防护 | 添加 `@media (hover: hover) and (pointer: fine)` |
| 进入/退出的时间对称 | 使退出比进入快 20–30% |
| CSS 变量拖拽动画 | 直接在元素上使用 `transform` |
| 拖拽时缺少 `setPointerCapture` | 添加指针捕获以实现可靠的跟踪 |
| Motion `x`/`y` 与手写的 `transform` 混合使用 | 为元素选择一个变换所有者 |
| 键盘操作触发动画 | 完全移除动画 |
| UI 元素动画时长 > 300ms | 减少到 150–250ms |
| 快速触发元素上使用关键帧 | 使用 CSS 过渡以实现可中断性 |
| 所有元素同时出现 | 添加交错延迟（项目间间隔 30–50ms） |

## 组件设计原则

- **优秀的默认值优于选项。** 大多数用户从不自定义。默认的缓动、时长和设计应该开箱即用就非常出色。
- **动态 UI 使用过渡而非关键帧。** 快速添加的元素（如通知、列表项）需要可中断的动画。关键帧在中断时从零重新开始；过渡可以平滑地重定向。
- **一致性。** 动画风格应与组件的个性相匹配。一个有趣的组件可以更有弹性。一个专业的仪表板应该干脆利落且快速。
- **处理不可见的边缘情况。** 当标签页隐藏时暂停计时器。用伪元素填充堆叠元素之间的间隙以保持悬停状态。在拖拽期间捕获指针事件。

## 调试动画

- **慢动作：** 临时将时长增加到 2–5 倍，或使用浏览器的动画检查器。检查颜色变化时机、缓动和变换原点。
- **逐帧检查：** 在 Chrome DevTools 的动画面板中逐步执行，以揭示协调属性之间的时序问题。
- **真实设备：** 对于触摸交互（抽屉、滑动手势），在物理设备上测试。Xcode 模拟器可用，但真实硬件更适合手势测试。
- **次日审查：** 用全新的视角你会注意到开发时遗漏的不完美之处。
