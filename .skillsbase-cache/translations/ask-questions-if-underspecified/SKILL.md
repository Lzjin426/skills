# 若需求不明确，请先提问

## 何时使用

当请求存在多种合理理解，或关键细节（目标、范围、约束、环境或安全性）不明确时，使用此技能。

## 何时不使用

当请求已足够清晰，或通过一次快速、低风险的探索性阅读即可填补缺失细节时，请勿使用此技能。

## 目标

提出避免错误工作所需的最少澄清问题；在必须回答的问题得到解答（或用户明确批准基于所述假设继续）之前，不要开始实施。

## 工作流程

### 1) 判断请求是否不明确

如果在探索如何执行工作后，以下部分或全部内容仍不清晰，则视为请求不明确：
- 明确目标（什么应改变，什么应保持不变）
- 明确“完成”标准（验收标准、示例、边界情况）
- 明确范围（哪些文件/组件/用户包含在内/排除在外）
- 明确约束（兼容性、性能、风格、依赖、时间）
- 识别环境（语言/运行时版本、操作系统、构建/测试运行器）
- 澄清安全性/可逆性（数据迁移、部署/回滚、风险）

如果存在多种合理理解，则假定其为不明确。

### 2) 先提出必须回答的问题（保持精简）

第一轮提出1-5个问题。优先选择能排除整个工作分支的问题。

使问题易于回答：
- 优化可扫描性（简短、编号的问题；避免段落）
- 尽可能提供多项选择选项
- 适当时建议合理的默认值（明确标记为默认/推荐选择；在列表中加粗推荐选择，或在代码块中呈现选项时，在块上方加粗“推荐”行，并在块内标记默认值）
- 包含快速路径响应（例如，回复 `defaults` 以接受所有推荐/默认选择）
- 在有用时包含低摩擦的“不确定”选项（例如，“不确定 - 使用默认值”）
- 将“需要知道”与“最好知道”分开，以减少摩擦
- 结构化选项，以便用户能以紧凑的决策方式回应（例如 `1b 2a 3c`）；用通俗语言重述所选选项以确认

### 3) 行动前暂停

在获得必须的答案之前：
- 不要运行命令、编辑文件或制定依赖于未知因素的详细计划
- 仅执行明确标记的、低风险的探索步骤，前提是该步骤不会使你承诺某个方向（例如，检查仓库结构、阅读相关配置文件）

如果用户明确要求你在没有答案的情况下继续：
- 将你的假设陈述为简短的编号列表
- 请求确认；仅在用户确认或纠正后继续

### 4) 确认理解，然后继续

一旦获得答案，用1-3句话重述需求（包括关键约束和成功标准），然后开始工作。

## 问题模板

- “在我开始之前，我需要知道：(1) ..., (2) ..., (3) .... 如果你不关心(2)，我将假定 ....”
- “应该是以下哪一种？A) ... B) ... C) ...（选择一个）”
- “你认为怎样才算‘完成’？例如：...”
- “是否有我必须遵守的约束（版本、性能、风格、依赖）？如果没有，我将以现有项目默认值为目标。”
- 使用带字母选项的编号问题和清晰的回复格式

```text
1) 范围？
a) 最小改动（默认）
b) 重构时涉及该区域
c) 不确定 - 使用默认
2) 兼容性目标？
a) 当前项目默认值（默认）
b) 同时支持旧版本：<指定>
c) 不确定 - 使用默认

回复格式：defaults（或 1a 2a）
```

## 反模式

- 不要提出可以通过快速、低风险的探索性阅读（例如，配置文件、现有模式、文档）来回答的问题。
- 如果紧凑的多项选择或是否问题能更快消除歧义，不要提出开放式问题。
