# 学术写作风格

将提供的信息转化为高质量的学术作业，匹配用户的自然写作风格，避免明显的人工智能模式，同时保持专业水准。

## 核心理念

生成自然流畅、易于阅读的内容，具备以下特点：
- 使用描述性标题进行清晰的章节组织
- 自然的主题推进，避免僵化的“首先……其次……最后”结构
- 根据作业类型适度使用第一人称视角
- 提供具体案例和细节，而非泛泛而谈
- 混合使用长短句，避免过度复杂
- 为目标语言（中文或英文）使用正确的标点符号

## 写作前准备

1.  **明确作业要求：**
    - 作业类型（技术分析、研究综述、案例研究等）
    - 目标语言（中文、英文或双语）
    - 预期长度或范围
    - 需要涵盖的具体主题或概念
    - 任何特殊要求

2.  **加载适当的参考资料：**
    - 对于中文作业：阅读 `references/chinese-examples.md`
    - 对于英文作业：阅读 `references/english-examples.md`
    - 始终阅读 `references/writing-guidelines.md` 以了解核心原则

3.  **评估个性化程度：**
    - 技术分析：更客观，极少使用第一人称
    - 研究综述：适度的个人声音
    - 案例研究：更高的个性化程度，可包含反思

## 写作流程

### 结构设计

创建描述性的章节标题，以预览内容，而非使用通用标签：
- 避免使用“引言” → 使用“Docker 与容器革命：一个实践视角”
- 避免使用“分析” → 使用“从繁琐到简洁：Spring Boot 如何改变 Java 开发”
- 避免使用“结论” → 使用“迁移生产数据库：一次零停机 PostgreSQL 切换的经验教训”

按自然主题流组织内容，让章节通过内容联系而非显式过渡词相互衔接。

### 段落构建

将信息整合成流畅的段落，而非列表。当信息自然形成列表时，将其嵌入散文中：

**避免：**
主要优势包括：
- 性能提升
- 成本降低
- 可扩展性增强

**推荐：**
此次优化带来了三个主要好处：性能显著提升，响应时间下降了 60%；通过更高效的资源使用降低了成本；架构获得了更好的可扩展性以应对未来增长。

### 过渡与流畅性

通过以下方式连接段落：
- **主题延伸：** 上一段落的最后一个概念在下一段落继续
- **自然对比：** 呈现对比观点，无需大量过渡词
- **隐含问题：** 回应内容引发的未言明的问题
- **章节分隔：** 使用章节划分来标志主要主题转换

避免使用机械的过渡词，如“然而”、“此外”、“另外”，让内容自然流动。

### 融入案例与细节

通过以下方式使写作具体化：
- 具体指标：“响应时间从 8 秒降至 2 秒”
- 真实案例：“Netflix 在几年内将其单体应用拆分为数百个微服务”
- 技术细节：“该查询涉及 7 个表连接并产生了 N+1 查询问题”
- 个人观察：“根据笔者的经验，这种方法适用于……”（谨慎使用）

### 语言校准

**对于中文写作：**
- 使用正确的中文标点：，。：“”
- 在适当处保留英文技术术语：“Spring Boot”、“Docker”
- 保持自然的中文句子节奏和流畅度
- 避免直接的英译中模式

**对于英文写作（雅思 6.0 水平）：**
- 优先使用常见词汇而非复杂词汇：用“use”而非“utilize”
- 通常保持句子在 30 个词以内
- 使用清晰、直接的句式结构
- 首次使用时定义缩写：“Object-Relational Mapping (ORM)”
- 混合句子长度以增强可读性

### 第一人称使用

策略性地使用第一人称视角：
- 描述实践经验：“笔者在项目中遇到过……” / “from my experience...”
- 表达有根据的观点：“我认为……” / “I found that...”
- 案例研究反思：“如果重新设计，我会……” / “looking back, I would...”

在以下情况保持客观性：
- 原理的技术解释
- 文献综述内容
- 纯粹的技术分析

## 质量验证

最终定稿前，验证：
- 不存在“首先……其次……最后”结构
- 极少使用项目符号（仅在绝对必要时）
- 段落通过内容自然连接
- 全文包含具体案例和细节
- 章节标题具有描述性和信息量
- 第一人称使用恰当且不过度
- 标点符号符合目标语言惯例
- 句子多样性（长短句混合）
- 语言避免了明显的人工智能痕迹
- 技术术语使用准确且一致

## 特殊考量

**对于双语作业（需要中英文版本）：**
- 独立撰写每个版本，而非直接翻译
- 根据每种语言的自然模式调整案例和措辞
- 在两个版本间保持技术准确性一致
- 根据每种语言语境适当调整正式程度

**对于技术分析：**
- 减少个人声音，增加客观性
- 专注于技术准确性和详细解释
- 使用来自真实系统或项目的具体案例
- 在可理解性与技术精确性之间取得平衡

**对于研究综述：**
- 将来源综合成叙述，而非简单罗列
- 展示观点的联系和演变
- 承认争议和不同视角
- 保持批判但平衡的语气

**对于案例研究：**
- 提供丰富的背景细节
- 包含遇到的具体挑战
- 反思经验教训（适合使用第一人称的地方）
- 平衡描述与分析

## 文件输出规范

### 输出目录规范

**推荐方法（遵循 Claude Code 官方标准）：**

将所有学术写作输出保存到 `outputs/<project-name>/writing/`：

```
outputs/
└── <project-name>/              # 项目名称（例如：cloud-computing-analysis）
    └── writing/
        ├── technical-analysis.md    # 技术分析报告
        ├── research-review.md       # 研究综述文档
        ├── case-study.md            # 案例研究报告
        └── project-documentation.md # 项目文档
```

**示例：**
```
outputs/
├── cloud-computing-analysis/
│   └── writing/
│       └── technical-analysis.md
├── ai-ethics-research/
│   └── writing/
│       └── research-review.md
└── database-optimization-case/
    └── writing/
        └── case-study.md
```

**替代方法（传统项目结构）：**

如果项目已有现有目录结构，也可以使用：

```
project-root/
└── docs/
    ├── technical-analysis.md
    ├── research-review.md
    └── case-study.md
```

### 输出文件列表

根据作业类型生成文档：

**技术分析：**
- `technical-analysis.md` - 技术分析报告

**研究综述：**
- `research-review.md` - 研究综述文档

**案例研究：**
- `case-study.md` - 案例研究报告

**项目文档：**
- `project-documentation.md` - 项目文档

### 文件命名规范

- 使用 kebab-case：`cloud-computing-technical-analysis.md`
- 需要时包含版本/日期：`research-review-v1.0.md`
- 使用描述性名称：`database-optimization-case-study.md`
- 如果是双语，请指定语言：`technical-analysis-en.md`，`technical-analysis-zh.md`

### 交付摘要

生成文档后，提供简要摘要：
- 文档类型和目标语言
- 字数统计和章节结构
- 涵盖的关键主题
- 应用的写作风格特点
- 文件保存位置确认
