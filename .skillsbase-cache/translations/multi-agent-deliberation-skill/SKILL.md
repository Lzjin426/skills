# 多智能体审议 (v1.1)

## 概述

使用此技能运行结构化的多智能体审议工作流，以获取质量优先的答案。

仅在辩论能带来价值时优先使用此技能。对于简单的事实性查询或低风险提示，单智能体已足够，请勿使用此技能。

## 工作流决策树

1. 验证激活门。
2. 定义决策目标和可测试标准。
3. 实例化辩论角色。
4. 在证据约束下运行有限轮次。
5. 使用固定评分表进行裁决。
6. 输出带有结构化状态和风险控制的最终答案。

## 步骤 1：验证激活门

使用多智能体辩论前，需满足全部三个条件：

1. 确认存在可争议空间。
2. 确认存在可测试标准。
3. 确认存在可证伪的证据链。

若任一条件缺失，请勿生成辩论角色。使用回退单智能体响应，并标记 `STATUS=GATE_FAILED`，同时列出缺失条件。

加载 `references/gate_and_controls.md` 以获取激活门检查清单和回退模式。

## 步骤 2：定义评估合约

在生成角色前设定以下项目：

1. 定义 `QUESTION` 和明确的 `CONSTRAINTS`。
2. 定义接受度量标准。
3. 定义来源质量策略。
4. 定义预算限制（轮次、令牌数、延迟）。

除非用户另行指定，否则使用以下默认评分表：

- 准确性：40%
- 完整性：25%
- 可追溯性：20%
- 反驳后的稳健性：15%

默认来源策略：

- 层级 A：主要来源（官方文档、标准、原始论文、权威数据集）。
- 层级 B：可靠的二次综合内容。
- 层级 C：观点性/未验证内容。
- `关键主张` 指可能改变结论方向、风险水平或推荐行动的主张。
- 关键主张需要至少一个层级 A 来源（高风险任务：需要两个独立的高质量来源）。

## 步骤 3：实例化辩论角色

默认创建五个角色：

1. `提议者-A`：提供候选答案 A 及证据图谱。
2. `提议者-B`：提供独立的候选答案 B 及证据图谱。
3. `批评者`：攻击两个候选答案并揭露隐藏假设。
4. `证据检查者`：验证主张与来源的一致性，并标记无支持的主张。
5. `裁判`：使用固定评分表进行评分和裁决。

隔离规则：

- A/B 必须在隔离环境中生成初稿。
- 在第一轮开始前，不得交叉阅读对方的推理草稿。

加载 `references/debate_playbook.md` 以获取可复用的提示块。

## 步骤 4：运行有限制审议

按明确计数运行各阶段：

1. 阶段 0（不计入辩论轮次）：独立提案（A/B）。
2. 第 1 轮：批评者交叉质询。
3. 第 2 轮：针对性反驳和证据更新。
4. 可选第 3 轮：仅当出现新证据或核心事实分歧仍未解决时进行。

轮次预算定义：

- 默认辩论轮次：2（第 1-2 轮）。
- 最大辩论轮次：3。
- 阶段 0 为强制步骤，不计入轮次预算。

停止规则：

- 裁判每轮输出 0-100 的总分。
- `分数增益 = 当前总分 - 前一轮总分`。
- 当连续两轮分数增益 < 1 分，或无新证据出现时停止。

## 步骤 5：裁决与决策

强制输出结构化决策：

1. 提供 `SCORE_A` 和 `SCORE_B`。
2. 选择获胜者或合并最终答案，并明确合并理由。
3. 输出最终答案，附带置信度和残余风险。
4. 输出可能改变决策的证据。
5. 输出无支持主张的处理方式（`dropped`、`pending_evidence` 或 `accepted_with_uncertainty`）。

若证据薄弱，切勿强制达成共识。

## 步骤 6：输出合约

始终输出以下结构：

```text
[STATUS]
DECIDED | MERGED | INSUFFICIENT_EVIDENCE | GATE_FAILED

[FINAL_ANSWER]
...

[SCORES]
SCORE_A: ...
SCORE_B: ...
SCORE_DELTA: ...

[CONFIDENCE_0_TO_100]
...

[KEY_EVIDENCE_MAP]
- claim_id -> source_id
...

[UNSUPPORTED_CLAIMS]
- claim_id
...

[DISPOSITION]
- claim_id: dropped | pending_evidence | accepted_with_uncertainty
...

[OPEN_RISKS]
...

[WHAT_WOULD_CHANGE_THE_ANSWER]
...

[NEXT_VERIFICATION_STEPS]
...
```

## 操作控制

1. 默认限制辩论轮次为 2；最大为 3（不包括阶段 0）。
2. 限制每个角色的响应长度，避免冗长偏见。
3. 所有关键主张必须提供来源 ID。
4. 拒绝无支持的关键主张，除非明确标记为 `accepted_with_uncertainty`。
5. 所有高风险任务在发布前必须经过人工审核。

加载 `references/gate_and_controls.md` 以获取故障模式和缓解规则。
