# Better Md Writer Core：文档意图

在起草任何实质性内容之前，使用此文件来确定文档需要完成的任务。

## 1. 明确任务，而非标签

当需求描述不明确时，仅识别那些会改变文档内容的缺失信息：
- 读者是谁？
- 文档的用途是什么？
- 读者阅读后应理解什么、决定什么或做什么？
- 文档主要用于解释、决策支持、执行、存档还是说服？
- 是否有格式、语气、长度或模板限制？

不要默认询问所有问题。只询问那些仍然未知且至关重要的信息。

## 2. 从六个维度解读任务

使用以下维度来塑造草稿：
- **受众熟悉度**：专家、混合、新手
- **目的**：解释、比较、决策、说服、记录、指导
- **可操作性**：信息性、决策支持性、执行驱动性
- **持久性**：临时笔记、可重用参考、长期文档
- **证据负载**：无、简要理由、明确证据
- **结构偏好**：段落、列表、表格、图表、混合

这些维度比过早地将需求套入通用模板更为重要。

## 3. 选择文档类型

将主要任务映射到一种类型：
- **笔记**：当主要任务是快速记录或低阻力整理时
- **知识库文章**：当主要任务是为后续重用提供持久性解释时
- **自述文件**：当主要任务是引导入门和快速上手时
- **提案**：当主要任务是比较选项并支持决策时
- **教程**：当主要任务是教授流程或概念时
- **操作手册**：当主要任务是操作执行时

如果两种类型似乎都合理，选择最符合读者下一步行动的那一种。

如果选择的类型是**知识库文章**，在起草前还需确定其主要子类型：
- **概念解释**：读者需要了解某物是什么、其表现形式以及边界在哪里
- **比较**：读者需要一个判断框架来区分多种机制、选项或类别
- **结果解读**：读者需要正确解读图表、输出、指标或分析结果
- **方法/流程**：读者需要理解输入-处理-输出链

这个子类型的选择应比作者的一时偏好更能控制文档的骨架结构。

## 4. 生成轻量级决策记录

在起草之前，将任务简化为一个紧凑的内部记录：

```text
类型：
知识库子类型：
受众：
目的：
可操作性：
持久性：
证据负载：
默认载体：
判断性问题：
视觉锚点：
仍需澄清：是/否
```

这是一个辅助工作的决策记录，不一定需要出现在最终文档中。

## 5. 默认保持轻量

不要仅仅因为任务听起来重要就启用研究、图表、认知路径设计或后处理。只有当它们明确能为当前读者和当前文档改善结果时，才启用它们。
