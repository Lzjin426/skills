# Better Md Writer Core：结构选择

在文档任务明确且已选定配置文件后使用本文档。

## 1. 选择主要载体

选择承载理解的主要形式：
- **段落主导**：用于解释、推理和连续阐述
- **列表主导**：用于简短并列的要点、步骤或清单
- **表格主导**：用于稳定的并列比较
- **图表主导**：仅当视觉呈现确实比文字解释更快时使用
- **混合形式**：当一种主要载体需要另一种形式的轻度支持时

不要让列表取代解释，也不要让表格取代判断。

## 2. 先结构，后句子

在润饰文字前先草拟大纲：
- 首先确定主要章节。
- 将价值最高或不确定性最高的章节放在前面。
- 如有需要，待核心内容稳定后再写引言和总结。
- 将相关的想法放在一起。

如果结构正确，句子层面的编辑会变得容易得多。如果结构错误，句子润饰也无法挽救文档。

对于知识库写作，不要仅从标题措辞开始。首先决定：
- 本文属于哪种子类型
- 本文必须回答的判断性问题是什么
- 读者需要什么样的视觉锚点

## 3. 优先优化理解

当主题涉及技术或抽象内容时：
- 在依赖术语之前，先用平实但精确的语言解释概念
- 在读者了解其作用后，再引入符号、变量名、API 名称或代码标识符
- 当能减少混淆时，尽早展示具体的产物、输出或视觉形态

使用平实但精确的语言。避免未经解释的行话和过于随意的教学语气。

## 4. 使用最合适的简单形式

根据内容功能选择：
- 使用**标题**表示可以独立存在的真实模块。
- 使用**短段落**，当一个紧凑的块已经能完成任务时。
- 使用**列表**，当项目是并列且简短时。
- 使用**表格**，当读者必须在相同维度上进行比较时。
- 使用**图表**，仅当其能实质性地降低解释成本时。

避免用装饰性标题或一堆小列表来分割页面。

对于知识库写作：
- 使用描述内容的语义化章节标题，而非口语化的钩子
- 避免标题读起来像叙述、指导或演示文稿文案
- 在章节开头，优先使用简短的引导段落，而非高亮的口号

## 5. 保持贯穿线

每个章节都应回答前一章节自然引出的一个问题。

对于解释性强的写作：
- 从读者最直接的问题开始
- 当能降低理解阻力时，按照“它是什么”→“它看起来像什么”→“我如何解读它”→“它如何工作”的顺序推进
- 合并薄弱章节，而不是产生许多内容很少的顶级标题

对于中等长度的知识库文章，`3-5` 个顶级章节通常比 `8-10` 个微小章节是更好的默认选择。

每篇知识库文章都应至少回答一个判断性问题，例如：
- 我如何识别这个东西？
- 我如何将其与相近概念区分开？
- 我如何正确解读这个输出？
- 我如何将其放回当前项目或工作流中？

每篇知识库文章还应建立至少一个视觉锚点，即使该视觉内容后来仅以候选标记的形式呈现。

低理解成本并不意味着随意的措辞。知识库文章应读起来像清晰的内部分考，而非口语化的课程记录。

## 6. 保持 Markdown 的可移植性

默认使用标准 Markdown：
- 仅在需要时使用清晰的标题
- 使用带语言标签的围栏代码块
- 使用有意义的链接文本
- 谨慎且有意识地使用强调
- 仅在兼容性明确时使用环境特定的语法

诸如编号标题或顶级分隔符之类的样式规则属于选定的配置文件或本地环境默认值，并非自动应用于每个文档。

当目标是 Fullstop 知识库或 Obsidian 库时，优先使用显式的分层标题编号，例如：
- `# 1. ...`
- `## 1.1 ...`
- `### 1.1.1 ...`

保持编号与实际大纲一致。如果编号变得尴尬，通常意味着大纲有误，应在继续起草前先简化大纲。

## 7. 仅在有益时拆分

考虑拆分为多个文档的情况：
- 内容包含多个可独立阅读的主题
- 单个文件变得难以浏览或维护
- 不同读者需要材料的不同部分

拆分时：
- 使每个文档自成一体
- 在文档间使用相对链接
- 为每个链接提供简短的内容说明
