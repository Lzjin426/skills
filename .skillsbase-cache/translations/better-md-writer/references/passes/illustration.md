# 更优 Markdown 写作流程：插图指南

插图是可选的，默认状态为“关闭”。

## 触发规则

如果一张视觉图能比额外的文字描述更快或更准确地建立正确的理解，则包含它。否则，不包含。

对于**知识库文章**，当主题是抽象的、基于图形的、基于模型的、基于用户界面的，或者仅凭文字难以想象时，倾向于包含更多视觉内容。

不要在文档以解释为主的主要写作流程中，单独做出最终的插图决定。应将插图审阅视为一项独立的认知任务。

## 强制插图审阅触发条件

当文档是知识库解释性文章，且满足以下任一条件时，必须进行独立的插图审阅：
- 主题是抽象的、基于模型的、基于图形的、基于用户界面的，或者难以想象的
- 草稿中包含 2 个或更多 `IMG-CANDIDATE` 标记
- 读者很自然地会问“这实际上看起来是什么样子？”
- 文档比较了多种机制、流程或模型系列

如果任何触发条件匹配，则插图审阅是强制性的，而非建议性的。

## 良好用例

- 流程、生命周期和分支路径
- 架构或分层关系
- 视觉上比文字更清晰的比较
- 能替代描述性文字的具体截图或参考图像
- 在正式解释之前展示“它看起来是什么样子”的示例
- 帮助读者将术语映射到实际产物的带注释示例

## 知识库文章倾向性

撰写知识库文章时，不要将视觉内容局限于结构图。在有用的情况下，优先采用混合形式：
- 真实截图或参考图像
- 示例输出
- 前后对比
- 带注释的图表
- 过程草图
- 简单的视觉类比

如果主题是读者可能会问“那实际上看起来是什么样子？”的内容，请尽可能早地用视觉方式回答这个问题。

## 工具选择

- 对于产品、地点、设备或用户界面截图等真实、具体的事物，使用网络图片
- 对于概念、结构、流程和关系，使用 `excalidraw-diagram`
- 对于笔记网络或知识关系，使用 `obsidian-canvas-creator`

## 工作流程

1.  在起草过程中，使用明确的候选占位符标记可能的图像位置，例如：
    - `<!-- IMG-CANDIDATE: excalidraw-diagram - 请求生命周期 -->`
    - `<!-- IMG-CANDIDATE: web - 仪表板设置面板 -->`
    - `<!-- IMG-CANDIDATE: example - 典型的 PDP 图表外观 -->`
    - `<!-- IMG-CANDIDATE: annotated - 标注坐标轴和主要曲线形状 -->`
2.  草稿结构稳定后，进行一次**专门的插图审阅**。
3.  让插图审阅决定：
    - 每个候选位置是否真的需要视觉内容
    - 视觉内容应承担什么任务
    - 哪种视觉类型最合适
    - 即使作者没有标记，是否应添加任何缺失的视觉内容
4.  仅在审阅步骤之后才制作视觉内容。
5.  只插入能降低解释成本的视觉内容。

## 专门的插图审阅

对于解释性强的知识库写作，建议使用独立的审阅代理或专门针对视觉内容的新审阅流程。

审阅者应检查整个草稿并回答：
- 读者会在哪里难以想象这个概念？
- 哪些部分在技术上是正确的，但没有视觉辅助阅读成本太高？
- 文档是否需要先有“它看起来是什么样子”的视觉图，然后再有结构图？
- 这是否最好通过真实示例、截图、输出样本、带注释的图表、过程草图或比较图来呈现？

此审阅步骤与制作图像不同。它的任务是制定一个**图像计划**。

## 图像计划输出

插图审阅应为每个批准的视觉内容生成一个简洁的计划：
- 在文档中的位置
- 为什么需要视觉内容
- 视觉类型
- 视觉内容应展示内容的简短描述
- 来源或制作路径：
    - 现有本地图像
    - 网络图片搜索
    - `excalidraw-diagram`
    - `obsidian-canvas-creator`

如果标记的候选位置不需要视觉内容，请明确说明，而不是习惯性地制作一个。

如果没有这个图像计划，满足强制触发条件的草稿应被视为不完整。

## 防护措施

- 不要强制规定图表配额
- 不要将插图视为默认的验收标准
- 移除那些重复文字而非辅助文字的视觉内容
- 对于知识库文章，插图不足通常比多一张有用的视觉图更失败
- 不要让主要写作代理成为解释性强的知识库文章图像必要性的唯一评判者
- 如果满足了强制触发条件，不要为了速度而跳过审阅步骤
